# 三项目 Favicon「先快后优」一次性优化计划书（2026-04-10）

## 1. 适用范围
本计划统一覆盖：
1. `Bookmark-Record-Recommend`
2. `Bookmark-Canvas`
3. `Bookmark-Backup`

目标：把“体感速度”放在第一优先级，同时保留后台闲置优化质量能力。

## 2. 核心策略（最终口径）
1. 第一阶段（前台）：只求快，先显示。
2. 第二阶段（后台闲置）：只做一次质量优化。
3. 初始建库：每个 URL 只优化一次。
4. 增量维护：新增/改 URL 后只优化一次。
5. 非增量场景（反复打开、反复切页）不重复做闲置优化。

## 3. 第一阶段：前台快路径
1. 渲染链路不等待“高质量完成”。
2. 命中缓存就直接显示。
3. 未命中就立即拉取可用图并显示。
4. 仅保留一个安全兜底门槛：`>=16x16`（避免无效垃圾图入库）。
5. 不在前台主路径执行透明/白底/复杂评分。

## 4. 第二阶段：后台闲置一次性优化
1. 触发时机：
初始建库与增量事件后进入 idle 队列。
2. 优化目标：
对同一 URL 尝试获得更高质量图（例如更大尺寸或更优来源）。
3. 执行次数：
每个 URL 每个策略版本仅一次（`upgradedOnce=true`）。
4. 覆盖规则：
只有“更好”才覆盖，不允许降级。
5. 结束状态：
标记完成后不再重复优化，直到 URL 发生变化或策略版本升级。

## 4.1 后台优化算法来源（已确认）
1. 复用你之前两次改动的算法思路（`ac95c84...`、`0538174...`）。
2. 复用范围：
候选拉取并发与超时控制、透明/白底/近白相关评分与候选选优策略。
3. 使用位置：
仅用于后台闲置优化队列，不进入前台首屏主路径。
4. 明确边界：
前台继续“先快后显”，后台才做质量升级；两条路径分离。

## 5. 数据模型（统一）
1. `favicon_store`：
`url -> iconData, minDimension, bytes, source, updatedAt, lastUsedAt`
2. `favicon_upgrade_state`：
`url -> upgradedOnce, upgradedAt, strategyVersion`
3. `favicon_reference`（可选但建议）：
`bookmarkId -> url`（用于垃圾清理判断是否仍被引用）
4. `failure_store`：
短 TTL 失败记录（防止失败风暴重试）

## 6. 增量维护规则
1. `onCreated(url)`：
写入引用，URL 入队，后台闲置优化一次。
2. `onChanged(url)`：
旧 URL 解除引用，新 URL 入队并优化一次。
3. `onRemoved`：
解除引用；若 URL 无引用则进入回收候选。
4. `onMoved` / 仅标题变更：
不触发图标优化。

## 7. 垃圾与空间控制
1. 建库过程中做轻清理：
清理无引用 + 失败过期项。
2. 建库阶段结束后做重清理：
按容量上限 + LRU 淘汰冷数据。
3. 清理优先级：
无引用 > 低质量 > 长期未用。

## 8. 三项目落地顺序
1. `Record-Recommend` 先落地：
验证“手动刷新/自动刷新/快照恢复”三条路径。
2. `Canvas` 第二批落地：
验证永久栏目、临时栏目与懒渲染兼容。
3. `Backup` 最后落地：
验证历史页与恢复链路不受影响。

## 9. 明确不做
1. 不做首次全量强制拉完再展示。
2. 不做前台主路径复杂评分。
3. 不做每次打开都重复闲置优化。

## 10. 验收口径
1. 首次/新增时图标出现明显更快（先显示）。
2. 同一 URL 后续打开不应重复长时间补图。
3. 后台优化完成后，质量逐步提升且不降级。
4. 大书签量下不会因全量建库导致明显卡顿或存储爆涨。
