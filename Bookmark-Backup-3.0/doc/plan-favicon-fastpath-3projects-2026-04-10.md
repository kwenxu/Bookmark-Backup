# 三项目 Favicon 链路与加速计划书（2026-04-10 更新版）

## 0. 重点与特点（先看这个）
1. 重点一：先把“慢”打下来，优先体感速度，不做大重构。
2. 重点二：三项目一套链路口径，减少后续维护分叉。
3. 重点三：不靠探针，不做运行中分支来回切换，行为可预测。
4. 重点四：保留质量底线（防降级），避免“越刷越糊”。
5. 特点：小改动、低风险、可分批落地、可随时回滚单项策略。

## 1. 适用范围
本计划覆盖以下三个项目：
1. `Bookmark-Backup`
2. `Bookmark-Record-Recommend`
3. `Bookmark-Canvas`

目标：降低首次安装与新增书签场景下的 favicon 体感延迟，并统一三项目链路行为。

## 2. 已确认最终决策
1. 分支判定不再使用探针。
2. 仅按语言桶判定分支：
中文（`zh`）走 `domestic`，非中文（`non_zh`）走 `overseas`。
3. 中文分支固定顺序：
`t3.gstatic.cn -> DuckDuckGo -> Google S2 -> cn.cravatar.com -> /_favicon`
4. 非中文分支固定顺序：
`DuckDuckGo -> Google S2 -> t3.gstatic.cn -> cn.cravatar.com -> /_favicon`
5. `/_favicon` 在两个分支都只作为末尾兜底。
6. 不做运行中动态切换分支，不做探针回切。

## 3. 当前已落地（3 项目一致）
1. 去探针切换：
`markNetworkBranchReevaluation`、`requestNetworkBranchReevaluation`、`_recordBranchProbeResult` 仅保留兼容壳子，不再切分支。
2. 分组并发拉取：
每轮按 `2+2+...` 小批次并发，而不是全串行。
3. 源级超时收紧：
不同源使用不同超时上限，避免慢源长期阻塞。
4. 三轮结构保留：
严格阈值轮 -> 放宽阈值轮 -> 终极兜底轮（16 仅兜底）。
5. 阈值参数已调整：
`strict=64`、`fallback=16`、`terminal=16`，当前配置下第三轮通常不会触发。

## 4. 为什么不做“全源全并发”
1. 请求洪峰大：
初装和批量新增时会显著放大并发请求总量。
2. 解析开销大：
返回结果越多，解码/尺寸读取/候选比较开销越大。
3. 收益不稳定：
全并发不一定更快，反而容易造成浏览器线程与网络抖动。

结论：当前采用“小批次并发 + 源级超时”是更稳妥的速度/资源平衡点。

## 5. 评分策略（当前与后续）
## 5.1 当前状态
1. 主决策链路已不再依赖透明/白底评分（已执行）。
2. 当前主决策改为：清晰度优先（最小边）+ 源顺序优先。
3. 旧评分函数仍保留为兼容代码，不参与主路径选优。

## 5.2 下一批可选优化（待确认）
1. 进一步降级评分：
仅保留清晰度门槛与防降级规则，不再做透明/白底偏好。
2. 主阈值下调评估：
评估是否将严格轮尺寸阈值进一步下调，以提升首次命中率。

## 6. 防降级规则（保留）
1. 新图标最小边 `<16`：拒绝写入。
2. 已有图标且新图更小：拒绝覆盖。
3. `16x16` 仅用于“填空”，不覆盖已有 `>=32` 结果。

## 7. 验收口径
1. 首次打开时，图标补齐速度明显快于改造前。
2. 新增书签后，图标补齐延迟明显缩短。
3. 多次打开同一页面，不应持续出现“从零补图”体感。
4. 旧高清图标不会被低清结果回刷覆盖。

## 9. 执行优先级（建议）
1. P0（已执行）：去探针 + 按语言分支固定链路 + 小批次并发 + 源级超时收紧。
2. P0（持续观察）：确认首次打开与新增补图的体感是否稳定，不出现反复“从零补图”。
3. P1（下一批）：评估评分进一步降级，仅保留清晰度门槛与防降级规则。
4. P1（下一批）：评估严格阈值是否可下调，以提升首轮命中率。
5. P2（后续）：如仍有慢点，再考虑更深层缓存建库方案，不在本轮展开。
