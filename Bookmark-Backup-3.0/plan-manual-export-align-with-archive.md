# 计划书：手动导出向主 UI 归档结构对齐

更新时间：2026-03-20

## 1. 本轮复查结论

这份计划已经收口到当前代码真实落地的版本：

1. `手动导出` 保留，但只用于 `本地下载 / 本地恢复 / 本地查看`。
2. `手动导出` 不再上传到 `云端1（WebDAV）` 和 `云端2（GitHub）`。
3. 主 UI 恢复继续保留第三类来源：`手动导出`。
4. `手动导出` 在恢复二级 UI 中继续区分：
   - `快照` -> `覆盖恢复`
   - `变化` -> `导入合并`
5. 云端归档继续由 `多版本 / 覆盖` 承担，不再为 `手动导出` 维护第二套云端归档。

一句话总结：

> `手动导出` 已经对齐到主 UI 恢复模型，但它现在是明确的本地来源，不再参与云端归档。

---

## 2. 为什么这样收口

### 2.1 云端已经有主归档体系

现有项目已经有两条稳定的云端主线：

- `多版本`
- `覆盖`

这两条已经覆盖常规备份、恢复和云端同步需求。

如果再把 `手动导出` 长期同步到云端，会额外带来：

- 重复归档
- 重复恢复入口
- WebDAV / GitHub 上传成本增加
- 维护复杂度继续上升

### 2.2 手动导出的核心场景本来就偏本地

`手动导出` 的价值主要是：

- 用户按需选择记录
- 用户按需选择 `simple / detailed / collection`
- 用户下载到本地
- 用户后续本地恢复或本地查看

这更像一个“本地导出包”体系，而不是云端常驻归档体系。

## 4. 主 UI 恢复的当前形态

### 4.1 一级来源

主 UI 恢复当前保留三类来源：

1. `多版本`
2. `覆盖`
3. `手动导出`

### 4.2 二级模式

三类来源共用统一语义：

| 一级来源 | 二级模式 | 可执行动作 |
|---|---|---|
| 多版本 | 快照 | 覆盖恢复 |
| 多版本 | 变化 | 导入合并 |
| 覆盖 | 快照 | 覆盖恢复 |
| 覆盖 | 变化 | 导入合并 |
| 手动导出 | 快照 | 覆盖恢复 |
| 手动导出 | 变化 | 导入合并 |

补充说明：

- `手动导出` 已经不再走旧的“直通逻辑”。
- `手动导出` 现在和 `多版本 / 覆盖` 一样，先进入统一的恢复二级 UI，再决定动作。

### 4.3 排序与索引

`手动导出` 当前复用了和 `多版本` 接近的排序逻辑：

- 有索引时优先按索引排序
- 无索引时按时间倒序
- 支持 `索引项 / 非索引项`
- `快照` 与 `变化` 共享同一条记录的时间、备注、指纹等元数据

## 5. 本地导出结构

### 5.1 根目录

本地导出的统一根目录：

- 中文：`书签备份/`
- 英文：`Bookmark Backup/`

### 5.2 手动导出父目录

本地手动导出统一放在：

- 中文：`书签备份/手动导出/`
- 英文：`Bookmark Backup/Manual Export/`

### 5.3 当前变化手动导出

当前变化下载时，产物落在：

- `书签备份/手动导出/当前变化/`

并包含：

1. 当前变化文件本体
2. `备份历史log.md`

当前行为：

- 会持续把当前变化导出记录并入本地 ledger
- 恢复时归入 `手动导出 -> 变化`
- 不归入 `快照`

### 5.4 备份历史全局导出 ZIP

`history_html` 的全局导出现在统一只生成 `ZIP 归档`，并会生成：

- ZIP 根目录
- 每条记录一个目录
- 每条记录目录下可包含：
  - `snapshot.html`
  - `changes-*.html / changes-*.json`
- 同目录下附带：
  - `备份历史log_开始_截止.md`

这就是当前主 UI 识别 `手动导出 -> 快照 / 变化` 的标准本地恢复包。

## 6. 云端策略

### 6.1 手动导出不上云

以下内容不再上传到云端：

1. `history_html` 的全局导出 ZIP
2. `history_html` 的当前变化手动导出
3. 与手动导出配套的 `备份历史log*.md`

### 6.2 云端恢复不扫描手动导出

以下路径不再进入云端恢复来源：

- `云端1（WebDAV）` 下的手动导出目录
- `云端2（GitHub）` 下的手动导出目录

也就是说：

- 主 UI 的 `手动导出` 来源现在是本地专用
- `云端1 / 云端2` 只保留 `多版本 / 覆盖`

## 8. 最终解释

这份计划在当前代码中的最终落点不是“手动导出全端统一上云”，而是：

1. `多版本 / 覆盖`
   - 继续承担常规归档与云端同步

2. `手动导出`
   - 作为第三类恢复来源保留
   - 但仅服务于本地下载、本地恢复、本地查看

因此当前最准确的描述是：

> `手动导出` 已经完成向主 UI 恢复模型的对齐，但它不再承担云端归档职责；云端归档继续由 `多版本 / 覆盖` 负责。
