# 恢复二级 UI 说明（当前实现）

更新时间：2026-03-20

## 1. 这份文件说明什么

这份文件不再保留早期“`手动导出` 不进入二级 UI”的旧方案。

当前代码里的真实行为是：

- `多版本 / 覆盖 / 手动导出` 都进入统一的恢复二级 UI
- `手动导出` 不再走旧的直通逻辑
- `手动导出` 现在明确是本地专用来源

---

## 2. 当前一级来源

主 UI 恢复当前存在三类来源：

1. `多版本`
2. `覆盖`
3. `手动导出`

补充：

- `手动导出` 仍然保留在左上角来源切换里
- 云端1 / 云端2 不再把手动导出纳入恢复来源

---

## 3. 当前二级切换

三类来源共用统一的二级切换：

- `快照`
- `变化`

动作语义如下：

| 一级来源 | 二级切换 | 可执行动作 |
|---|---|---|
| 多版本 | 快照 | 覆盖恢复 |
| 多版本 | 变化 | 导入合并 |
| 覆盖 | 快照 | 覆盖恢复 |
| 覆盖 | 变化 | 导入合并 |
| 手动导出 | 快照 | 覆盖恢复 |
| 手动导出 | 变化 | 导入合并 |

说明：

- `快照` 页最终都落到 `覆盖恢复`
- `变化` 页最终都落到 `导入合并`

---

## 4. 当前手动导出规则

### 4.1 来源限制

`手动导出` 目前只接受本地来源：

- 本地手动导出文件夹
- 本地手动导出 ZIP
- 本地当前变化手动导出
- 本地备份历史全局导出 ZIP

### 4.2 云端限制

以下来源不会再进入 `手动导出`：

- `云端1（WebDAV）`
- `云端2（GitHub）`

原因很简单：

- 云端归档已经由 `多版本 / 覆盖` 承担
- 手动导出更适合作为本地按需导出包
- 这样可以避免重复归档和重复恢复入口

### 4.3 索引与排序

`手动导出` 当前支持：

- `索引项`
- `非索引项`

排序原则：

1. 有索引时按索引优先
2. 无索引时按时间倒序

---

## 5. 当前交互原则

1. `多版本 / 覆盖 / 手动导出` 都走统一二级 UI
2. `手动导出` 也显示 `快照 / 变化`
3. `当前变化` 手动导出只进入：
   - `手动导出 -> 变化`
4. `history_html` 的全局导出 ZIP 可以同时供：
   - `手动导出 -> 快照`
   - `手动导出 -> 变化`

---

## 6. 当前状态结论

旧计划里这两条已经作废：

- `手动导出` 不显示二级切换
- `手动导出` 继续走完全直通

当前实际实现是：

- `手动导出` 显示二级切换
- `手动导出` 只在本地生效
- 云端恢复不再承接手动导出

一句话总结：

> 恢复二级 UI 现在是“三类来源统一二级语义”，但 `手动导出` 已被明确收口为本地专用来源。
